# DESIGN_ANSWERS.md

**Q1: How would you design the system to support many different types of voice notes?**

The `PREDEFINED_SCHEMAS` dictionary in `structuring_engine.py` is the schema registry. Adding a new note type requires only adding a new key to this dict — the LLM prompt is constructed dynamically from it, and `note_type` is stored as an open string (not an enum) so new types round-trip through the system without code changes. Approach B's dynamic structure captures types that no one anticipated when writing the predefined schemas. For a production system, I would move the schema registry from a hardcoded dict into the JSON store itself (a `schemas.json` file), allowing operators to add, edit, or deprecate note types via an admin API without redeployment. Each schema entry would include a version number and an `active` flag.

**Q2: Predefined vs AI-generated schemas — advantages and disadvantages?**

Predefined schemas are queryable, consistent, and debuggable — a `task.scheduled_time` field can be filtered in a list query, indexed in a database, and sent to a calendar integration. The cost is rigidity: you must anticipate every use case, and real-world voice notes rarely fit cleanly into any single type. AI-generated schemas are flexible and future-proof but inconsistent — two identical observations might produce `{"issue": "aphids"}` in one call and `{"pest": "aphids", "pest_type": "insect"}` in another. The hybrid approach in AgriMemo gets the best of both: Schema A for reliable downstream integration, Schema B as a safety net that captures everything Schema A misses. The `note_type` field, derived from Schema A, is the primary queryable classifier.

**Q3: How would you scale to thousands of users per minute?**

The current synchronous-per-request design becomes the bottleneck first. The key change is to make intake asynchronous: the upload endpoint saves the audio to object storage (Cloudflare R2, since we are already in the Cloudflare ecosystem) and publishes a message to Cloudflare Queues. Worker processes (separate services) consume the queue, run Deepgram and the LLM calls, and write results to a real database (Cloudflare D1 or PostgreSQL). The HTTP endpoint returns a `note_id` immediately and the client polls for completion. FastAPI can then handle many more concurrent uploads since it is no longer blocked by 2-second Deepgram calls. The JSON file store is replaced with D1 (SQLite at the edge). AI Gateway's caching means repeated identical transcripts (duplicates) hit the cache and cost nothing. Deepgram's batch API or streaming API handles volume spikes without per-request round trips.

**Q4: Challenges in noisy environments?**

Background noise is the primary challenge. STT models trained on clean speech degrade significantly in environments with wind, machinery, crowd noise, or rain. Mitigations in order of implementation cost: (1) the `low_confidence` flag already built into AgriMemo surfaces the problem to the user — they know when to re-record; (2) Deepgram's `smart_format` and noise-reduction parameters help at the API level; (3) server-side audio preprocessing via pydub/ffmpeg: a high-pass filter at 80Hz removes low-frequency hum (tractors, pumps), and a noise gate suppresses constant background noise between words; (4) in a future mobile app, push-to-talk (what AgriMemo already uses via the upload model) is fundamentally more reliable than always-on voice activity detection because the user controls exactly when the microphone is capturing. The upload-only model in AgriMemo is actually an advantage here — users can re-record and re-upload if the first attempt produces a low-confidence result.
