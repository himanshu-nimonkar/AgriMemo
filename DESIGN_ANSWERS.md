# Tellia Intern Tech Test Design Answers

### How would you design the system to support many different types of voice notes

The current architecture addresses the challenge of supporting diverse note types by using a two pronged large language model approach. Instead of forcing every spoken word into a single rigid box we process the transcript through two separate paths at the same time. The first path attempts to map the content to a set of predefined categories that are common in agricultural work like tasks or field observations. The second path is a dynamic extraction where the AI creates its own structure based on the specific details it finds in the text. This means that if a user records something completely new like a project note about a specific equipment failure the dynamic schema will capture those unique details without needing a specific developer to define a new template beforehand. In a more mature version we could add a routing layer that identifies the intent of the note first and then selects a specialized prompt or schema to improve accuracy even further.

### What are the advantages and disadvantages of predefined schemas versus AI generated schemas

Predefined schemas are excellent when you need high reliability and predictable data for secondary systems like a calendar or a task manager. They ensure that every task always has a title and a due date which makes building dashboards much simpler for developers. The downside however is their rigidity because they often fail to capture nuanced information that does not fit the template. AI generated schemas solve this by being incredibly flexible and able to grow as the user needs change. They can extract entities like specific bugs in a field or precise chemical names that might not be in a standard list. The trade off is that parsing this dynamic data can be more complex for downstream software because the field names might vary. Using both concurrently as we do in this prototype provides the best of both worlds by giving us stable data for core features while still preserving the full context through AI discovery.

### How would you scale this system if thousands of users were sending voice notes every minute

To scale a system like this to handle thousands of requests every minute we would need to move away from the current single server and file based storage model. The first step would be introducing a message queue like RabbitMQ or Amazon SQS to handle the background processing of transcription and structuring. This would allow us to ingest audio files very quickly and acknowledge the request while the actual heavy lifting happens on a separate group of worker nodes. We would also replace the local JSON file storage with a distributed database like PostgreSQL or a globally distributed key value store to ensure data is safe and reachable. Using a content delivery network and an object store like Amazon S3 for the raw audio files would prevent the main application server from being overwhelmed by large file transfers. This decoupled architecture ensures that a spike in users recording notes would only grow the task queue rather than crashing the system itself.

### What challenges would arise when capturing voice in noisy environments

Capturing clear audio in a noisy farm or industrial setting is one of the biggest hurdles for voice first systems. Farm machinery or wind noise can easily muffle a user's voice and lead to poor transcription accuracy. We can address this on both the client and the server side. On the mobile device we can use hardware based noise cancellation or specialized recording settings to prioritize human speech frequencies. Once the audio reaches the server we can apply digital signal processing filters to reduce background hum before sending the data to the transcription engine. Another critical strategy is to use the confidence scores provided by the speech to text service. If the AI is not sure what the user said we can flag the note for review or ask the user for a quick clarification while they are still in the field. This ensures that a single gust of wind does not result in a permanent error in the structured database.
